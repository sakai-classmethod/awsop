# 要件ドキュメント

## はじめに

このドキュメントは、awsop コマンドの zsh シェル補完機能を拡張し、プロファイル名だけでなくコマンドラインオプションも補完できるようにする機能の要件を定義します。現在の実装ではプロファイル名のみが補完対象ですが、ユーザビリティ向上のため、`-`で始まるオプションも補完候補として提供します。

## 用語集

- **Shell Completion（シェル補完）**: シェルが Tab キーを押した際に、入力候補を自動的に提示・補完する機能
- **awsop**: AWS 認証情報を 1Password 経由で管理する CLI ツール
- **zsh**: Z Shell。本機能の対象シェル
- **compdef**: zsh の補完関数を登録するコマンド
- **\_describe**: zsh の補完候補を表示する組み込み関数
- **Option（オプション）**: コマンドラインオプション（`-h`, `--help`など）
- **Profile（プロファイル）**: AWS 設定ファイルに定義されたプロファイル名
- **Context-aware Completion（文脈認識補完）**: 入力文脈に応じて適切な補完候補を提示する機能

## 要件

### 要件 1

**ユーザーストーリー:** CLI ユーザーとして、`-`を入力後に Tab キーを押したときに利用可能なコマンドラインオプションが表示されるようにしたい。そうすることで、オプションを暗記しなくても発見して使用できる。

#### 受け入れ基準

1. WHEN ユーザーが`awsop -`と入力して Tab キーを押す THEN システムは利用可能なすべての短縮オプション（例：`-h`, `-s`, `-u`）を表示する
2. WHEN ユーザーが`awsop --`と入力して Tab キーを押す THEN システムは利用可能なすべての長形式オプション（例：`--help`, `--show-commands`, `--unset`）を表示する
3. WHEN ユーザーが`awsop -r`と入力して Tab キーを押す THEN システムは`--region`に補完するか、`-r`で始まるオプションを表示する
4. WHEN オプション補完を表示する THEN システムは各オプションの簡潔な説明を含める
5. WHEN ユーザーが部分的なオプションを入力して Tab キーを押す THEN システムは一致するオプションのみをフィルタして表示する

### 要件 2

**ユーザーストーリー:** CLI ユーザーとして、プロファイル名の補完が以前と同様に動作し続けるようにしたい。そうすることで、既存のワークフローが中断されない。

#### 受け入れ基準

1. WHEN ユーザーが`awsop `（`-`なし）と入力して Tab キーを押す THEN システムは利用可能な AWS プロファイル名を表示する
2. WHEN ユーザーが`awsop prod`と入力して Tab キーを押す THEN システムは"prod"に一致するプロファイルをフィルタして表示する
3. WHEN ユーザーが`awsop -s `と入力して Tab キーを押す THEN システムはプロファイル名を表示する（オプションではない）
4. WHEN プロファイル補完がトリガーされる THEN システムは`~/.aws/config`ファイルからプロファイルを読み取る
5. WHEN AWS 設定ファイルが存在しない THEN システムはシェルを壊すことなくエラーを適切に処理する

### 要件 3

**ユーザーストーリー:** CLI ユーザーとして、補完システムがオプションとプロファイルのどちらを表示すべきかを賢く判断するようにしたい。そうすることで、文脈に基づいた関連性の高い提案を得られる。

#### 受け入れ基準

1. WHEN 現在の単語が`-`で始まる THEN システムはオプション補完のみを提供する
2. WHEN 現在の単語が`-`で始まらない THEN システムはプロファイル補完のみを提供する
3. WHEN オプションが値を必要とする（例：`--region`） THEN システムはそのオプションの後にオプション補完を提供しない
4. WHEN `--source-profile`または`--output-profile`の後で補完する THEN システムはプロファイル名補完を提供する
5. WHEN ユーザーがすでにプロファイル名を入力している THEN システムは後続の引数に対してオプション補完を提供する

### 要件 4

**ユーザーストーリー:** 開発者として、オプションリストが保守可能で実際の CLI 実装と同期されるようにしたい。そうすることで、補完が常に現在利用可能なオプションを反映する。

#### 受け入れ基準

1. WHEN 補完関数が生成される THEN システムは CLI で定義されたすべてのオプションを含める
2. WHEN オプションに短縮形と長形式の両方がある THEN システムは両方を補完リストに含める
3. WHEN オプションの説明が提供される THEN システムは`option:description`の形式で表示する
4. WHEN CLI オプションが更新される THEN 補完関数は容易に更新可能である
5. WHEN シェルラッパーを生成する THEN システムは説明付きの完全なオプションリストを含める

### 要件 5

**ユーザーストーリー:** CLI ユーザーとして、補完が zsh の既存の補完機能とシームレスに動作するようにしたい。そうすることで、慣れ親しんだ補完動作を使用できる。

#### 受け入れ基準

1. WHEN zsh の matcher-list 設定を使用する THEN システムは設定されている場合、大文字小文字を区別しないマッチングを尊重する
2. WHEN zsh の部分一致を使用する THEN システムはオプションとプロファイルの両方で部分文字列マッチングをサポートする
3. WHEN 複数の補完候補が存在する THEN システムはそれらを読みやすい形式で表示する
4. WHEN ユーザーが補完候補を循環する THEN システムは zsh の標準的な補完動作に従う
5. WHEN 補完関数がロードされる THEN システムは他の zsh 補完関数と競合しない

### 要件 6

**ユーザーストーリー:** CLI ユーザーとして、オプション補完に役立つ説明が含まれるようにしたい。そうすることで、ヘルプテキストを参照しなくても各オプションの機能を理解できる。

#### 受け入れ基準

1. WHEN オプション補完を表示する THEN システムはオプション名とその説明を表示する
2. WHEN オプションに短縮形がある THEN システムは短縮形と長形式の両方に説明を含める
3. WHEN 説明が長すぎる THEN システムは表示用に適切に切り詰める
4. WHEN オプションが表示される THEN システムは読みやすさのために説明を整列する
5. WHEN ユーザーがオプションを選択する THEN システムはオプション名のみを挿入する（説明は含めない）

### 要件 7

**ユーザーストーリー:** 開発者として、シェルラッパー生成が後方互換性を保つようにしたい。そうすることで、既存ユーザーのセットアップが更新後も動作し続ける。

#### 受け入れ基準

1. WHEN `--init-shell`コマンドが実行される THEN システムは有効な zsh 関数を出力する
2. WHEN 生成された関数が source される THEN システムはエラーを発生させない
3. WHEN 古い補完関数が置き換えられる THEN システムは同じ関数名`_awsop`を維持する
4. WHEN ラッパー関数が更新される THEN システムは既存の`awsop()`関数の動作を保持する
5. WHEN ユーザーが`--init-shell`を再実行する THEN システムは更新された補完関数を出力する
