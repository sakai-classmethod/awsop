# 要件ドキュメント

## はじめに

awsop ツールのバージョン管理を改善し、Single Source of Truth の原則に従って`pyproject.toml`を唯一のバージョン情報源とする。現在、バージョン情報が複数箇所にハードコードされており、手動同期が必要で同期忘れのリスクがある。`importlib.metadata`を使用してバージョン情報を動的に取得することで、保守性と信頼性を向上させる。

## 用語集

- **awsop**: AWS 認証情報を管理するコマンドラインツール
- **pyproject.toml**: Python プロジェクトのメタデータと依存関係を定義する設定ファイル
- **importlib.metadata**: Python の標準ライブラリで、インストール済みパッケージのメタデータにアクセスするためのモジュール
- **Single Source of Truth (SSOT)**: データや情報が単一の場所にのみ定義され、他の場所はそれを参照する設計原則
- **フォールバック**: 主要な方法が失敗した場合に使用される代替手段

## 要件

### 要件 1

**ユーザーストーリー:** 開発者として、バージョン情報を一箇所でのみ管理したい。そうすることで、バージョン更新時の手動同期作業を削減し、同期忘れによるバグを防止できる。

#### 受入基準

1. WHEN pyproject.toml の version フィールドが更新される THEN システムは他のファイルを変更することなく新しいバージョンを反映しなければならない
2. THE システムは pyproject.toml をバージョン情報の唯一の情報源として使用しなければならない
3. THE システムは **init**.py ファイル内でバージョンをハードコードしてはならない
4. WHEN 開発者がバージョンを更新する THEN 開発者は pyproject.toml のみを編集しなければならない

### 要件 2

**ユーザーストーリー:** 開発者として、importlib.metadata を使用してバージョン情報を動的に取得したい。そうすることで、インストール済みパッケージのメタデータから正確なバージョン情報を得られる。

#### 受入基準

1. WHEN システムがバージョン情報を取得する THEN システムは importlib.metadata.version()を使用しなければならない
2. WHEN パッケージがインストールされている THEN システムは pyproject.toml で定義されたバージョンを返さなければならない
3. THE システムは Python 3.11 以上の標準ライブラリ機能のみを使用しなければならない
4. WHEN バージョン取得が成功する THEN システムは文字列形式のバージョン番号を返さなければならない

### 要件 3

**ユーザーストーリー:** ユーザーとして、パッケージが未インストールの状態でもツールが適切に動作してほしい。そうすることで、開発環境でのテストや直接実行時にもエラーが発生しない。

#### 受入基準

1. WHEN パッケージが未インストールの状態でバージョン情報が要求される THEN システムは適切なフォールバック値を返さなければならない
2. WHEN importlib.metadata.version()が PackageNotFoundError を発生させる THEN システムは例外をキャッチして処理しなければならない
3. WHEN フォールバック処理が実行される THEN システムは "unknown" または明確な代替バージョン文字列を返さなければならない
4. WHEN フォールバック値が使用される THEN システムは正常に動作を継続しなければならない

### 要件 4

**ユーザーストーリー:** ユーザーとして、`awsop -v`コマンドでバージョン情報を確認したい。そうすることで、使用しているツールのバージョンを簡単に把握できる。

#### 受入基準

1. WHEN ユーザーが `awsop -v` を実行する THEN システムは "awsop X.Y.Z" 形式でバージョンを表示しなければならない
2. WHEN ユーザーが `awsop --version` を実行する THEN システムは `-v` と同じバージョン情報を表示しなければならない
3. WHEN バージョンコマンドが実行される THEN システムは動的に取得したバージョン情報を表示しなければならない
4. WHEN バージョン表示後 THEN システムは正常終了しなければならない

### 要件 5

**ユーザーストーリー:** 開発者として、開発環境と本番環境の両方で同じバージョン取得メカニズムが動作してほしい。そうすることで、環境による動作の違いを心配せずに開発できる。

#### 受入基準

1. WHEN パッケージが `uv tool install -e .` で編集可能モードでインストールされる THEN システムは正しいバージョンを返さなければならない
2. WHEN パッケージが通常モードでインストールされる THEN システムは正しいバージョンを返さなければならない
3. WHEN 開発環境でバージョンが取得される THEN システムは本番環境と同じバージョン文字列を返さなければならない
4. WHEN 異なるインストール方法が使用される THEN システムは一貫したバージョン取得動作を提供しなければならない

### 要件 6

**ユーザーストーリー:** 開発者として、既存の CLI インターフェースとの互換性を維持したい。そうすることで、既存のユーザーやスクリプトに影響を与えずに内部実装を改善できる。

#### 受入基準

1. WHEN バージョン管理の実装が変更される THEN システムは既存の `-v` オプションの動作を維持しなければならない
2. WHEN バージョン管理の実装が変更される THEN システムは既存の `--version` オプションの動作を維持しなければならない
3. WHEN ユーザーがバージョンコマンドを実行する THEN システムは以前と同じ出力形式を提供しなければならない
4. WHEN 内部実装が変更される THEN システムは外部 API の互換性を保持しなければならない
